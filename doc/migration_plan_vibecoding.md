# OFDM Host 迁移计划（VibeCoding 版）

## 给非技术同学的 1 分钟摘要

这份文档可以先这样理解：
- 我们不会一下子推翻旧系统，而是分阶段升级。
- 旧版采集能力继续可用，先把底层能力拆出来，再逐步换新界面。
- 每一阶段都要求“可运行、可回退、可验证”，避免一次性大改带来的风险。

当前进度（截至本次更新）：
- Phase 1 已完成：后台服务骨架和通信协议已打通。
- Phase 2 已完成：串口开关、触发检测、记录落盘和实时事件流已打通。
- Phase 3 已完成：新前端已接入实时波形、日志过滤、离线分析与导出策略。
- Phase 3 第二批已完成：分析结果导出、波形时间窗/缩放、日志高亮已接入。
- Phase 3 第二批增强已完成：导出目录配置与自动/固定量程切换已接入。
- Phase 3 收尾已完成：导出命名模板/重名策略、固定量程双轴独立配置、日志高流量性能优化已接入。
- Phase 4 step1 已完成：新增 preflight、自检启动脚本、发布构建脚本、发布指南与 UI 启动前路径自检。

你最关心的结论：
- 日常采集暂不受影响，仍可继续使用现有主程序。
- 迁移期间会保留回退路径，出现问题可切回旧版。
- 每次大改都会自动留版本记录，便于追踪和回滚。

## 0. 目标与范围

### 0.1 目标
将当前 Python + PyQt6 上位机升级为：
- Flutter 桌面前端（Windows 优先）
- Python 核心服务（串口采集、解析、落盘、统计）
- 前后端通过本地 IPC 通信（建议 JSON Lines）

### 0.2 本期范围（第一阶段）
本次先做“前端迁移 + 核心保留”，不做核心算法重写：
- 保留现有 Python 核心逻辑：serial_reader.py / process_data.py / fix_data_format.py
- Flutter 实现新 UI、交互、状态管理、波形展示
- Python 作为独立后台进程，为 Flutter 提供实时数据流与控制接口

### 0.3 暂不纳入
- 串口能力改写为 Dart 原生插件
- 核心算法迁移到 Go/Rust
- 多平台发布（先 Windows 可用）

## 2. 迁移难度评估（Flutter 前端 + Python 核心）

## 总体结论
- 总体难度：中等（3.2/5）
- 可行性：高
- 风险主要在 IPC 协议设计、长时稳定性与打包联调，不在 UI 绘制本身

### 2.1 模块难度分解
| 模块 | 难度 | 说明 |
|---|---:|---|
| Flutter UI 重建（M3 + 自适应） | 2/5 | 页面结构清晰，属于工程实现问题 |
| 状态管理迁移（Riverpod） | 2/5 | 状态明确，易拆分为 provider |
| 实时波形渲染 | 2/5 | 可用 fl_chart/custom_painter 快速实现 |
| Python 核心服务化（IPC） | 4/5 | 最大迁移点，需定义稳定协议与错误处理 |
| 打包与分发（Flutter + Python） | 4/5 | 需处理进程生命周期、路径、权限与版本 |
| 回归测试与行为一致性 | 3/5 | 需对齐旧版边界行为（断包、乱码、触发词） |

### 2.2 关键风险
- 协议不稳定：事件定义不清会导致前后端频繁返工
- 长时运行风险：串口抖动、缓存膨胀、UI 卡顿
- 打包复杂度：Python 解释器/依赖随包分发策略
- 行为回归：旧版细节逻辑在迁移中被“简化”导致结果不一致

### 2.3 风险缓解策略
- 先协议后页面：先定事件模型，再接 UI
- 先跑通最小闭环：连接 -> 触发 -> 记录 -> 停止 -> 分析
- 回归样本集：用 historydata 目录真实日志做对比测试
- 全链路日志：Flutter 与 Python 分别记录通信日志

## 3. 目标架构（建议）

### 3.1 进程结构
- Flutter Desktop App
  - 展示层、交互层、状态管理层
- Python Core Service（独立进程）
  - 串口管理、实时解析、文件落盘、统计计算

### 3.2 通信方式（建议优先级）
1. JSON Lines over stdin/stdout（首选，最轻量）
2. 本地 WebSocket（次选，调试友好）

建议先用 JSON Lines，后续如需多客户端接入再升级 WebSocket。

### 3.3 事件协议（v1 草案）

请求（Flutter -> Python）：
- app.init
- serial.list_ports
- serial.open
- serial.close
- record.start
- record.stop
- file.process
- app.shutdown

响应/事件（Python -> Flutter）：
- app.ready
- serial.ports
- serial.connected
- serial.disconnected
- trigger.detected
- stream.text
- stream.hex
- metric.offset_delay
- metric.packet_loss
- record.started
- record.stopped
- process.result
- error

统一字段建议：
- id: 请求 ID
- type: 消息类型
- ts: 毫秒时间戳
- payload: 业务数据
- code/message: 错误码与描述

## 4. VibeCoding 迁移计划（分阶段）

## 原则
- 小步快跑，每个阶段都可运行、可演示、可回退
- 每周都有可见成果，不做“大爆炸式重写”

### 阶段 0：基线冻结（2-3 天）
目标：明确“旧版行为标准”，防止迁移偏航。

交付物：
- 功能基线清单（当前版本能力清单）
- 典型日志样本集（正常/异常/边界）
- 回归对比指标（统计值误差、触发时序、丢包计数）

### 阶段 1：协议与骨架（3-5 天）
目标：打通 Flutter 与 Python 进程通信。

交付物：
- Python core_service.py（仅实现 list_ports + heartbeat）
- Flutter 壳工程（桌面窗口 + Riverpod 基础）
- 协议文档 v1（请求/响应/错误码）

验收标准：
- Flutter 可展示端口列表
- 异常可被 UI 感知并提示

### 阶段 2：采集闭环 MVP（5-7 天）
目标：先跑通核心业务链路。

范围：
- 打开/关闭串口
- 等待触发后记录
- 实时文本流显示
- 录制停止与文件落盘

验收标准：
- 能稳定完成 30 分钟连续采集
- 与旧版对比，记录文件结构一致

### 阶段 3：数据可视化与统计（4-6 天）
目标：补齐体验层功能。

范围：
- offset 波形实时绘制
- offset/delay 指标卡片
- 丢包告警显示
- process_data 统计结果页面化

验收标准：
- 统计结果与旧版误差在可接受范围（建议 <= 1e-6）
- UI 无明显卡顿（目标 60fps，最低可接受 30fps）

### 阶段 4：工程化与发布（4-7 天）
目标：可交付、可安装、可回滚。

范围：
- Flutter Windows 打包
- Python 依赖随包分发策略
- 崩溃日志与诊断信息
- 发布文档与运维手册

验收标准：
- 一键启动
- 默认路径与权限可用
- 卸载/升级流程可验证

### 阶段 5：灰度替换（3-5 天）
目标：降低切换风险。

范围：
- 新旧版本并行试运行
- 问题回收与修复
- 最终切换与旧版冻结

验收标准：
- 关键功能 0 阻断问题
- 关键指标达标后再全量替换

## 7. 验收与回归策略

### 7.1 功能验收
- 串口连接/断连、异常恢复
- 触发录制、自动/手动保存
- offset/delay 解析与显示
- 丢包检测与计数
- 历史文件统计分析

### 7.2 非功能验收
- 连续运行稳定性（建议 8 小时）
- 内存占用无持续泄漏
- UI 交互响应在可接受范围

### 7.3 对比基线
- 用同一批 historydata 样本，比较：
  - offset/delay 均值
  - 收敛时间
  - 丢包计数
- 偏差超阈值必须给出解释或修复
